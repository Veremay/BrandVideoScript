系统流程可以从 **Brief 上传 → 多视角解析 → 脚本创作与节点化反馈 → 协商准备输出** 来描述。

## 1. Brief 上传与项目初始化

创作者上传品牌 Brief

## 2. 多智能体初始解析

系统通过 Coordinator Agent 调度不同 Agent 进行解析。

**Brand Agent** 负责理解品牌方视角：
它从 Brief 中提取明确要求，以及调用Tavily工具检索品牌方资料、Brand Wiki的品牌知识，推测品牌方可能没有写清楚但后续会在意的隐性需求。

**Audience Agent** 负责在已有 Persona 前提下模拟观众对脚本的反应：
它**不读取 Brief**；目标观众 **Persona** 由独立的数据分析管线生成（见 `technical_plan.md` §4.3），可根据平台语境和内容类型推断，后续可对接真实数据分析服务。

**Expert Agent** 负责提供内容创作与商业视频经验：
它结合品牌合作视频创作知识、同领域优质案例和脚本结构经验（留一个工具接口在这，系统会提供相关知识库），判断 Brief 对创作的影响，并为后续修改方案提供依据。

这些结果会被写入 State，但**不同 Agent 只读取自己需要的上下文，避免信息污染**。例如：**Audience 不读 Brief 与品牌需求**；**Brand 不读 Persona**；Expert 读取双方结构化输出；Coordinator 可读全部 artifact 并按视角控制对用户的披露（见 `technical_plan.md` §4、§16）。

## 3. 生成初始节点图

系统将解析结果转化为节点图。节点按照 IBIS 结构组织：

* **Issue**：需要讨论或解决的问题
  例如“产品露出是否会破坏内容自然性？”
* **Position**：针对问题的不同立场
  例如“强化产品卖点”或“弱化硬广感”
* **Argument**：支持或反对某一立场的理由
  例如“品牌希望提高转化”或“观众可能反感突兀植入”

不同图标可以代表 Issue 的来源，例如：

* Brand Agent 提出的品牌需求问题
* Audience Agent 提出的观众接受度问题
* Expert Agent 提出的创作策略问题
* Creator 手动提出的问题
* 真实品牌反馈转化的问题（这个可以考虑让Brand Agent再次解析）

## 4. 创作者进入主工作区

主界面以 **Coordinator Agent Chat** 为统一入口。创作者不需要分别管理复杂的 Agent 调度，而是通过 Coordinator 与系统交流。

界面主要包含：

* **Script Editor**
* **Node Graph**
* **Persona 面板入口**
* **Coordinator Agent Chat**
* **Negotiation Preparation / References 输出面板**（MVP先不做）

其中 Script Editor 和 Node Graph 是主要面板，可以互相切换。

## 5. Script Editor 中进行脚本创作

创作者在 Script Editor 中手动撰写或修改脚本。系统始终保留创作者的主导权，不会直接替代创作者完成全部改写。

脚本可以采用表格结构

当创作者修改脚本时，系统会更新 Shared State，并触发相关 Agent 对变动部分进行分析。

## 6. 基于脚本内容触发多视角反馈

创作者可以选中脚本中的某段内容，并通过 Coordinator 发起问题，Coordinator Agent 会根据问题调用对应 Agent。

Agent 的反馈不会只是聊天文本，而是可以被结构化为新的节点（如果有必要）：

* 新 Issue
* 新 Position
* 新 Argument
* 对已有节点的补充或修改

例如，创作者写了一段较强产品卖点的脚本后，系统可能生成：

* Issue：产品介绍是否过于硬广？
* Position A：保留强卖点表达
* Position B：改为剧情化植入
* Argument：品牌需要清晰传达卖点；但观众可能降低观看兴趣。

## 7. Node Graph 中组织冲突与权衡

Node Graph 用来呈现脚本创作中的关键问题。

创作者可以在节点图中：

* 查看节点
* 修改节点内容
* 删除节点
* 增加节点
* 标记某些 Position 为「待考虑」（TO BE CONSIDERED），表示创作者采纳该立场并希望在修改方案中落实

节点图的作用是帮助创作者看清：当前脚本中有哪些品牌需求、观众接受度和创作策略之间的冲突。

## 8. Persona 面板维护观众视角

Persona 面板作为入口存在于主界面中。系统会根据**数据分析接口**生成初始 Persona，但创作者可以随时修改。

Persona 可能包含：

* 目标观众描述
* 年龄范围
* 兴趣偏好
* 内容消费习惯
* 对广告植入的敏感点
* 观看动机
* 可能的反感点

当 Persona 被修改后，Audience Agent 的反馈逻辑也会相应更新，并影响后续观众视角节点。

## 9. 真实品牌反馈输入

当创作者将这个Script分享给品牌方后（生成share的链接），品牌方只能看到script表格本身，其他看不见。品牌方可以把反馈写入script editor中的feedback一列。

系统会将真实反馈转化为新的项目状态：

Brand Agent 解析反馈中的：

* 明确修改要求
* 新增约束

Expert Agent 基于这些反馈提出新的修改策略。

系统也会把真实品牌反馈转化为新的 IBIS 节点，和原有节点形成对照。

## 10. Expert Agent 生成修改方案

创作者在 Node Graph 中将采纳的 **Position** 加入 **TO BE CONSIDERED** 列表；点击列表中的「生成修改方案」后，Expert Agent 基于这些立场生成脚本修改方案。

修改方案融入 **Script Editor** 脚本表格：有改动的单元格内直接展示 cell-level **diff**（原文删除线 → 建议文案），创作者可对每一处选择 **Accept** 或 **Reject**，再通过工具栏整体或部分应用到脚本。

当创作者通过 Coordinator 请求修改建议时，Expert Agent 也可生成不同方案（逻辑同上）。

这些方案不是简单给一个“最佳答案”，而是提供多个可选方向，例如：

* 保守方案：更贴近品牌需求
* 平衡方案：兼顾品牌露出和内容自然性
* 创作者主导方案：保留创意表达，但提供解释理由
* 观众友好方案：降低广告感，提高观看接受度

每个方案需要说明：

* 修改了什么
* 解决哪个 Issue
* 牺牲了什么
* 适合什么沟通场景
* 可能被品牌方质疑的点
* 如何回应品牌方

## 11. Negotiation Preparation 输出（暂时取消）

当创作者准备与品牌方沟通时（点击按钮），系统生成 Negotiation Preparation，生成好之后出现一个popout的弹窗。

它基于当前脚本、节点图、品牌需求、观众 Persona 和专家建议，输出一份协商准备材料，包括：

* 当前脚本的核心设计意图
* 已满足的品牌需求
* 尚存在分歧的问题
* 每个争议点的解释理由
* 可接受的让步空间
* 不建议让步的创作底线
* 面对品牌方质疑时的回应话术
* 推荐优先沟通的问题顺序

这个输出帮助创作者在正式沟通前明确：哪些地方可以改，哪些地方需要坚持，以及如何解释自己的创作选择。

## 12. References 输出（先不做）

References 和 Negotiation Preparation 位于同一个输出面板中，但可以互相切换。

References 主要展示系统用于支持方案的参考依据，例如：

* 同领域优质视频案例
* 品牌营销知识
* 视频创作原则
* 观众接受度相关依据
* Expert Agent 使用的外部知识

它的作用是增强创作者解释方案时的可信度，而不是只给出主观建议。

## 13. 最终结果

整个流程的最终产出不是单一脚本，而是一组面向协商的创作资产：

* 一版持续迭代的脚本
* 一张结构化节点图
* 一个可编辑 Persona
* 多个脚本修改方案
* 一份协商准备材料
* 一组支持方案的 References

系统的核心价值是：
**在品牌方正式 review 或沟通之前，帮助创作者提前识别潜在冲突、理解品牌隐性需求、预测观众反应，并准备好可解释、可协商的脚本方案。**
